メール添付ファイルの安全な暗号化方法

パスワード付きZIPファイルを送り、その後にパスワードを別メールで送る手法は、日本では古くから「PPAP（Password付きZIPファイルを送ります、Passwordを送ります、Aん号化（暗号化）協議会、Protocol）」と呼ばれてきました。

しかし現在、多くの公的機関、企業では、この手法は「セキュリティ効果がほぼなく、むしろリスクを高める」として廃止が進んでいます。

パスワード付きファイル＋平文メールでパスワードを送る方式の危険性

（大学・公的機関でも廃止が進む理由）

① 暗号化ファイルもパスワードも "同じ経路" で盗まれる

一番の理由は、「泥棒が同じ道を通っている」ことに気づいていない点です。

  • 1通目のメール（ファイル）： 攻撃者に盗聴されたとします。
  • 2通目のメール（パスワード）： 同じネットワーク、同じ設定で送られているため、同様に盗聴される可能性が極めて高いです。

玄関の鍵を閉めて（1通目）、その鍵を玄関のマットの下に置いた（2通目）のを、泥棒に後ろから見られているような状態です。公的なセキュリティ指針を策定しているIPA（独立行政法人 情報処理推進機構）なども、この「経路の同一性」がセキュリティ上の意味をなさないことを指摘しています。

公的機関・企業での警告：「同じ経路を通る限り盗聴対策にならない」と指摘されています。

悪意ある第三者がメールを盗み見られる状況なら、暗号化ファイルもパスワードも両方盗まれ、容易に解読されることになります。

参考: NTT West, NTT

② ZIPの暗号方式が弱く、短いパスワードは数分で破られる

多くのWindows環境では ZipCrypto という古い弱い方式が使われます。これはブルートフォース攻撃に弱く、短いパスワードなら数分〜数日で解読可能と報告されています。

大学の研究データや個人情報の保護には十分とは言えません。

③ 誤送信時に「パスワード別送」がほぼ意味をなさない

PPAP方式は「もし暗号化ファイルを間違って送っても、パスワードを送らなければ大丈夫」という考えが元になっています。

しかし実際には：

  • 誤送信した相手にパスワードメールも送ってしまう可能性が高い
  • 両方同じタイミング・同じ宛先に送るため誤送信リスクが連動

と公的資料でも指摘されています。初心者にわかりやすく言うと、「間違いに気づかないまま鍵まで送ってしまう」ため、安全性を確保できません。

④ 政府・金融庁・大企業が "明確に廃止" を宣言

政府（内閣府・デジタル庁）や金融庁は、「パスワード付きZIPのメール送付は望ましくない」として、中央省庁・企業に対して廃止を強く推奨しています。

「パスワード付きファイルのメール送付は基本的には行うべきではない」
— 金融庁（2025年資料）

大学（.ac.jp）でも同様のガイドラインに従い、PPAPを禁止・縮小している例が増加しています。

初心者向けまとめ：なぜ危険なのか

  1. 暗号化ファイルとパスワードが同じ経路で盗まれるため安全ではない。
  2. ウイルスチェックができず、マルウェア拡散の手法として悪用されている。
  3. 誤送信時に"鍵も一緒に送ってしまう"ので、情報漏えい防止策として機能しない。

代わりに何を使うべきか（大学・公的機関が推奨）

  • クラウドストレージ（Box / OneDrive / Google Drive）によるリンク共有
  • メール自体の暗号化（S/MIME / PGP）
  • 専用のファイル転送サービス
  • 公開鍵暗号方式によるファイル暗号化（このツール）

公開鍵暗号方式の仕組み

PPAP（パスワード別送）の最大の問題は、「鍵（パスワード）」そのものを相手に送らなければならない点にありました。これを根本から解決するのが、「公開鍵暗号方式」です。

1. 仕組み： 「南京錠」と「自分だけの鍵」

公開鍵暗号方式は、セットで作られた「公開鍵（Public Key）」と「秘密鍵（Private Key）」という2つの鍵を使います。

よく例えられるのが、「南京錠（公開鍵）」と「その南京錠を開ける物理的な鍵（秘密鍵）」の関係です。

  1. 鍵の準備（受信者）： 受信者は「開いた南京錠（公開鍵）」と「自分だけが持つ鍵（秘密鍵）」のペアを作ります。
  2. 南京錠を配る： 受信者は、ファイルを送ってほしい人に「開いた南京錠（公開鍵）」を渡します。これは誰に盗まれても構いません。
  3. ロックして送る（送信者）： 送信者は、もらった南京錠でファイルにガチャンとロックをかけます。一度閉めると、送信者本人でも二度と開けられません。
  4. 自分の鍵で開ける（受信者）： ロックされたファイルを受け取った受信者は、手元に隠し持っている「秘密鍵」でロックを解除します。

2. なぜPPAPより圧倒的に安全なのか？

PPAPとの決定的な違いは、「暗号を解くための鍵（秘密鍵）を、一度もネットワーク上に流さない」という点にあります。

鍵を盗まれるリスクがない

PPAPでは、パスワード（共通鍵）をメールで送るため、そこを狙われれば終わりでした。公開鍵方式では、ネットワークを流れるのは「誰でも手に入れられる南京錠（公開鍵）」だけです。泥棒が南京錠だけを盗んでも、中身を開けることは絶対にできません。

数学的な「一方通行性」

公開鍵暗号は、数学的な「解くのは簡単だが、逆算するのは極めて困難」という性質（トラップドア関数）に基づいています。たとえば、非常に大きな素数同士の掛け算は一瞬で計算できますが、その答えから元の素数を見つける（素因数分解する）には、スーパーコンピュータを使っても何年もかかるという性質を利用しています。

3. 実社会での活用例

この仕組みは、現代のインターネットの安全性（情報の機密性・真正性）を支える基盤となっています。

  • HTTPS（Webサイト閲覧）： ブラウザで「鍵マーク」が出る通信は、この仕組みでデータを保護しています。
  • S/MIME（電子署名・暗号化メール）： 大学や官公庁で、メールのなりすまし防止や秘匿化に使われます。
  • 電子署名： 「秘密鍵」でデータに印をつけ、「公開鍵」で検証することで、「間違いなく本人が送ったものである」という証明（ハンコのような役割）も可能です。

このツール（WebCrypto Exchange KH）について

このツールは「公開鍵暗号方式」の添付ファイル用の暗号化をブラウザ上だけで実装しています。

実装上の補足

JavaScript 標準機能の活用： 公開鍵暗号を扱うための Web Crypto API（window.crypto.subtle）が利用可能になっています。本ツールはこの標準機能を利用して鍵生成・暗号化・復号を行っています。

ローカルファイル処理： 本アプリはブラウザの Blob / File API を使ってローカルファイルを読み書きします。ファイルはブラウザ内で処理され、外部サーバーに送信されることはありません。

GitHub Pages での静的ホスティング： GitHub Pages のような静的ホスティング環境ではサーバーサイドの受信や処理機能が存在しないため、本アプリはサーバーにファイルや鍵を送信することがそもそもできません。

ハイブリッド暗号化の仕組み

このツールはハイブリッド暗号化方式を採用しています：

ハイブリッド暗号化の仕組み

  1. AES-256-GCM暗号化：大容量ファイルを高速に暗号化するため、ランダムに生成したAES鍵を使用してファイルを暗号化します
  2. RSA-OAEP鍵暗号化：生成したAES鍵を秘密鍵の持ち主のみが復号できるよう、公開鍵でRSA暗号化します

この仕組みにより、画像、動画、圧縮ファイルなど、サイズ制限なく安全にファイルを暗号化できます。

このツールでは、受取人が鍵ペアを生成し、公開鍵のみを送り手に送信します。送り手は受け取った公開鍵でファイルを暗号化し、受取人は秘密鍵でそれを復号します。

注意（簡易ツールとしての位置づけ）

このアプリは簡易的で、専門的に整備された暗号化ツール（商用製品や組織向けの堅牢なソリューション）にはおおよそ及びません。

ですが、誤解を恐れずに言えば、PPAPの手法（パスワード付きZIP＋平文でのパスワード送付）を続けるよりは、実践的に安全性の改善に寄与します。

運用上の安全性や監査要件が重要な場合は、より専門的なツールや組織のポリシーに従ったソリューションを利用してください。
