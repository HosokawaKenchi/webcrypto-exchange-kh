Safe Encryption of Email Attachments

Sending a password-protected ZIP file and then emailing the password in a separate message is a practice known in Japan as "PPAP" (Password-protected ZIP → Password sent → Ango-ka / Encryption → Protocol).

Today, many government agencies and companies have abolished PPAP, recognising that it provides virtually no security benefit and in fact increases risk.

Why "Password-ZIP + Plain-text Password" is Dangerous

(The reason universities and public agencies are phasing it out)

① Both file and password travel the same route — so both can be stolen

The core problem is simple: the thief watches the same road twice.

  • Email 1 (file): Potentially intercepted by an attacker.
  • Email 2 (password): Sent over exactly the same network and servers — equally vulnerable.

It's like locking your front door (Email 1), then leaving the key under the mat right in front of the same door (Email 2), while the thief watches from behind. Japan's IPA (Information-technology Promotion Agency) explicitly warns that sending both over the same channel provides no meaningful protection.

Warning from public authorities: "Sending both over the same route provides no eavesdropping protection."

If an attacker can intercept your email, both the encrypted file and the password are stolen simultaneously — and decryption is trivial.

References: NTT West, NTT

② ZIP's encryption is weak — short passwords are cracked in minutes

Most Windows environments use ZipCrypto, a legacy algorithm widely considered broken. Brute-force attacks can crack short passwords in minutes to days. This is inadequate for research data or personal information.

③ "Send password separately" fails at exactly the wrong moment

The original rationale was: "Even if you send the file to the wrong person, they can't open it without the password." In practice:

  • Users often send the password email immediately after, to the same wrong recipient.
  • Both emails share the same misdirection risk — the mistake is correlated.

In plain terms: "You send the lock and the key to the wrong person without realising it."

④ Governments, financial regulators, and large companies have explicitly banned PPAP

Japan's Cabinet Office, Digital Agency, and Financial Services Agency have all declared that sending password-protected ZIPs by email is not recommended and have pushed ministries and enterprises to discontinue the practice.

"Password-protected file email attachment should fundamentally not be used."
— Japan Financial Services Agency (2025)

Summary: Why It's Dangerous (Plain Language)

  1. The file and its password travel the same route — both can be intercepted.
  2. Antivirus tools cannot scan inside encrypted ZIPs — attackers exploit this to spread malware.
  3. Mis-sent files come with their keys, so "sending password later" does not prevent disclosure.

Recommended Alternatives

  • Cloud storage (Box / OneDrive / Google Drive) with share links
  • Encrypted email (S/MIME / PGP)
  • Dedicated secure file transfer services
  • Public-key file encryption (this tool)

How Public-Key Cryptography Works

PPAP's fundamental problem was that you must send the key (password) to the other party. Public-key cryptography solves this at the root.

1. The Padlock Analogy

Public-key cryptography uses a pair of keys: a public key and a private key.

  1. Create keys (Receiver): The receiver generates an "open padlock" (public key) and a unique physical key (private key).
  2. Share the padlock: The receiver gives the open padlock to anyone who wants to send a file. Stealing a padlock is useless without the key.
  3. Lock and send (Sender): The sender clicks the padlock shut on the file. After locking, even the sender cannot open it.
  4. Unlock with private key (Receiver): The receiver uses their secretly-held private key to unlock the file.

2. Why It's Far More Secure Than PPAP

The decisive difference: the decryption key (private key) never travels over any network.

No risk of the key being stolen

In PPAP, the password (shared key) is emailed — that one interception point is fatal. In public-key cryptography, only the "open padlock" (public key) travels across the network. Stealing a padlock is completely useless for opening the locked content.

Mathematical one-way function

Public-key cryptography relies on a mathematical trapdoor function — easy in one direction, computationally infeasible in reverse. For example, multiplying two very large prime numbers is instant, but factoring the product (given only the result) would take supercomputers years.

3. Real-world Applications

  • HTTPS (web browsing): The padlock icon in your browser uses this mechanism to protect data.
  • S/MIME (email encryption): Used by universities and government agencies to prevent impersonation and protect email content.
  • Digital signatures: The private key signs data; the public key verifies it — proving authenticity without revealing the signer's secret.

About This Tool (WebCrypto Exchange KH)

This tool implements public-key file encryption entirely in the browser.

Technical Notes

Web Crypto API: The tool uses window.crypto.subtle — a standard browser API for key generation, encryption, and decryption.

Local file processing: Files are read and written using the browser's Blob / File API. No file content is ever sent to an external server (unless you do so yourself).

Static hosting (GitHub Pages): Because GitHub Pages is a static host with no server-side processing, the app is technically incapable of uploading files or keys to any server.

Hybrid Encryption

The tool uses hybrid encryption:

  1. AES-256-GCM encryption: A randomly generated AES key encrypts the file at high speed, supporting files of any size.
  2. RSA-OAEP key encryption: The AES key is then encrypted with the recipient's public key, so only the holder of the private key can recover it.

This makes it possible to securely encrypt files of any size — images, videos, archives, and more.
