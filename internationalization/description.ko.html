이메일 첨부파일의 안전한 암호화

비밀번호로 보호된 ZIP 파일을 보낸 후, 비밀번호를 별도의 이메일로 전송하는 방식을 "PPAP"(Password-protected ZIP → Password sent → Encryption → Protocol)이라고 합니다.

현재 많은 정부 기관과 기업들이 PPAP을 폐기했습니다. 이 방식이 실질적인 보안 효과를 제공하지 않으며, 오히려 위험을 높인다는 것을 인식했기 때문입니다.

왜 "비밀번호 보호 ZIP + 평문 비밀번호"가 위험한가

(대학과 공공 기관이 이를 포기하는 이유)

① 파일과 비밀번호 모두 동일한 경로로 전송 — 둘 다 도청될 수 있음

핵심 문제는 간단합니다: 도둑이 같은 도로를 두 번 지나갑니다.

  • 이메일 1 (파일): 공격자에 의해 도청될 수 있습니다.
  • 이메일 2 (비밀번호): 정확히 동일한 네트워크와 서버로 전송 — 취약성이 동일합니다.

현관문을 잠근 후(이메일 1), 같은 현관 앞 매트 아래에 열쇠를 놓아두는 것(이메일 2)과 같으며, 도둑이 뒤에서 지켜보고 있습니다. 일본의 IPA (정보처리진흥기구)는 명시적으로 동일한 경로로 둘 다 전송하는 것이 의미 있는 보호를 제공하지 않음을 경고합니다.

공공 기관의 경고: "동일한 경로로 전송하면 도청 방지 효과가 없습니다."

공격자가 이메일을 도청할 수 있다면, 암호화된 파일과 비밀번호 모두 동시에 도용되며 — 복호화는 자명합니다.

참고: NTT West, NTT

② ZIP 암호화는 약함 — 짧은 비밀번호는 분 단위로 파괴됨

대부분의 Windows 환경에서는 ZipCrypto, 광범위하게 손상된 것으로 간주되는 레거시 알고리즘을 사용합니다. 전수 조사 공격은 짧은 비밀번호를 분에서 일 단위로 파괴할 수 있습니다. 이는 연구 데이터나 개인 정보 보호에 부적절합니다.

③ "비밀번호 별도 전송"이 정확히 잘못된 순간에 실패함

원래 근거는: "파일을 잘못된 사람에게 보낸다 해도, 비밀번호를 보내지 않으면 열 수 없습니다." 실제로는:

  • 사용자가 종종 비밀번호 이메일을 바로 직후에, 동일한 잘못된 수신자에게 보냅니다.
  • 두 이메일이 동일한 방향 착오 위험을 공유 — 실수가 상관관계 있음.

더 간단히 말하면: "당신이 인식하지 못한 채 잠금장치와 열쇠를 잘못된 사람에게 보냅니다."

④ 정부, 금융 규제 기관, 대기업이 명시적으로 PPAP 금지

일본의 내각부, 디지털청, 금융청은 모두 이메일로 비밀번호 보호 ZIP을 보내는 것이 권장되지 않음을 선언했으며 중앙 부처와 기업에 이 관행을 중단할 것을 권장했습니다.

"비밀번호 보호 파일 이메일 첨부는 기본적으로 사용하지 않아야 합니다."
— 일본 금융청 (2025)

요약: 왜 위험한가 (일반인 설명)

  1. 파일과 비밀번호가 동일한 경로로 여행 — 둘 다 도청될 수 있습니다.
  2. 안티바이러스 도구가 암호화된 ZIP 내부를 스캔할 수 없음 — 공격자가 이를 악용하여 멀웨어를 배포합니다.
  3. 잘못 보낸 파일이 열쇠와 함께 옵니다, 그래서 "비밀번호 나중에 보내기"는 공개를 방지하지 않습니다.

권장 대안

  • 클라우드 스토리지 (Box / OneDrive / Google Drive) 공유 링크 포함
  • 암호화된 이메일 (S/MIME / PGP)
  • 전용 보안 파일 전송 서비스
  • 공개 키 파일 암호화 (이 도구)

공개 키 암호화의 작동 원리

PPAP의 기본적인 문제는 상대방에게 열쇠(비밀번호)를 보내야 한다는 것이었습니다. 공개 키 암호화는 이를 근본적으로 해결합니다.

1. 자물쇠 비유

공개 키 암호화는 한 쌍의 열쇠를 사용합니다: 공개 키와 개인 키.

  1. 열쇠 만들기 (수신자): 수신자는 "열린 자물쇠"(공개 키)와 고유한 물리적 열쇠(개인 키)를 만듭니다.
  2. 자물쇠 공유: 수신자는 파일을 보내고 싶은 사람에게 열린 자물쇠를 줍니다. 자물쇠를 도용하는 것은 열쇠 없이는 쓸모가 없습니다.
  3. 잠금 및 전송 (발신자): 발신자가 파일에 자물쇠를 잠급니다. 잠근 후, 발신자도 이를 열 수 없습니다.
  4. 개인 키로 잠금 해제 (수신자): 수신자는 비밀리에 보관 중인 개인 열쇠를 사용하여 파일의 잠금을 해제합니다.

2. PPAP보다 훨씬 더 안전한 이유

결정적인 차이: 복호화 열쇠(개인 키)는 어떤 네트워크도 통과하지 않습니다.

열쇠를 도용당할 위험 없음

PPAP에서 비밀번호(공유 열쇠)가 이메일로 전송되므로 — 그 한 지점의 도청이 치명적입니다. 공개 키 암호화에서는 "열린 자물쇠"(공개 키)만 네트워크를 통과합니다. 자물쇠를 도용하는 것은 잠긴 내용을 열기에 완전히 쓸모가 없습니다.

수학적 일방향 함수

공개 키 암호화는 수학적 함정문 함수에 의존합니다 — 한 방향으로는 쉽지만, 역산은 계산적으로 불가능합니다. 예를 들어, 두 개의 매우 큰 소수를 곱하는 것은 즉시이지만, 곱한 결과만 주어졌을 때 원래 소수를 인수분해하는 것은 슈퍼컴퓨터가 수년이 걸립니다.

3. 실제 응용 사례

  • HTTPS (웹 브라우징): 브라우저의 자물쇠 아이콘은 이 메커니즘을 사용하여 데이터를 보호합니다.
  • S/MIME (이메일 암호화): 대학 및 정부 기관에서 사칭 방지 및 이메일 내용 보호에 사용됩니다.
  • 전자 서명: 개인 열쇠가 데이터에 서명하고; 공개 열쇠가 이를 검증합니다 — 서명자의 비밀을 드러내지 않고 진정성 증명합니다.

이 도구 소개 (WebCrypto Exchange KH)

이 도구는 공개 키 파일 암호화를 전적으로 브라우저에서 구현합니다.

기술 노트

Web Crypto API: 도구는 window.crypto.subtle을 사용합니다 — 열쇠 생성, 암호화, 복호화를 위한 표준 브라우저 API입니다.

로컬 파일 처리: 파일은 브라우저의 Blob / File API를 사용하여 읽고 쓰여집니다. 파일 내용은 절대 외부 서버로 보내지지 않습니다 (당신이 직접 하지 않는 한).

정적 호스팅 (GitHub Pages): GitHub Pages가 서버 측 처리가 없는 정적 호스트이기 때문에, 앱은 기술적으로 파일이나 열쇠를 어떤 서버에도 업로드할 수 없습니다.

하이브리드 암호화

도구는 하이브리드 암호화를 사용합니다:

  1. AES-256-GCM 암호화: 임의로 생성된 AES 열쇠가 고속으로 파일을 암호화하며, 모든 크기의 파일을 지원합니다.
  2. RSA-OAEP 열쇠 암호화: AES 열쇠가 수신자의 공개 열쇠로 암호화되므로, 개인 열쇠 보유자만 이를 복구할 수 있습니다.

이는 모든 크기의 파일을 안전하게 암호화할 수 있게 합니다 — 이미지, 비디오, 아카이브 등.

부인 성명

이 앱은 경량 도구이며 상업적으로 또는 조직적으로 유지되는 암호화 솔루션의 견고성과 일치하지 않습니다.

즉, PPAP (비밀번호 보호 ZIP + 평문 비밀번호 이메일)에 비해 측정 가능한 보안 개선을 제공합니다. PPAP이 대안이라면, 이 도구가 더 나은 선택입니다.

감사 추적 가능성 또는 공식 보안 보증이 필요한 환경의 경우, 조직의 정책과 일치하는 전문적으로 관리되는 솔루션을 사용하시기 바랍니다.
