Sichere Verschlüsselung von E-Mail-Anhängen

Das Versenden einer passwortgeschützten ZIP-Datei und anschließend das Senden des Passworts in einer separaten E-Mail ist eine in Japan als "PPAP" bekannte Methode (Password-protected ZIP → Password sent → Ango-ka / Encryption → Protocol).

Heute haben viele Regierungsbehörden und Unternehmen PPAP abgeschafft, da sie erkannt haben, dass es praktisch keinen Sicherheitsvorteil bietet und das Risiko tatsächlich erhöht.

Warum "Passwort-ZIP + Plaintext-Passwort" gefährlich ist

(Der Grund, warum Universitäten und öffentliche Behörden da abkommen)

① Datei und Passwort reisen beide über denselben Weg — beide können abgefangen werden

Das Kernproblem ist einfach: Der Dieb beobachtet dieselbe Straße zweimal.

  • E-Mail 1 (Datei): Möglicherweise von einem Angreifer abgefangen.
  • E-Mail 2 (Passwort): Wird über exakt dasselbe Netzwerk und dieselben Server gesendet — gleich anfällig.

Es ist wie das Verschließen der Haustür (E-Mail 1) und dann das Verlegen des Schlüssels unter der Matte direkt vor der Tür (E-Mail 2), während der Dieb von hinten zuschaut. Japans IPA (Information-technology Promotion Agency) warnt ausdrücklich, dass das Senden beider über denselben Kanal keinen bedeutungsvollen Schutz bietet.

Warnung der Behörden: "Das Senden beider über dieselbe Route bietet keinen Abhörschutz."

Wenn ein Angreifer Ihre E-Mail abhören kann, werden sowohl die verschlüsselte Datei als auch das Passwort gleichzeitig gestohlen — und die Entschlüsselung ist trivial.

Referenzen: NTT West, NTT

② ZIPs Verschlüsselung ist schwach — kurze Passwörter werden in Minuten geknackt

Die meisten Windows-Umgebungen verwenden ZipCrypto, einen Legacy-Algorithmus, der allgemein als beschädigt gilt. Brute-Force-Angriffe können kurze Passwörter in Minuten bis Tagen knacken. Dies ist unzureichend für Forschungsdaten oder personenbezogene Informationen.

③ "Passwort separat senden" scheitert genau im falschen Moment

Die ursprüngliche Begründung war: "Selbst wenn Sie die Datei an die falsche Person senden, können sie sie ohne das Passwort nicht öffnen." In der Praxis:

  • Benutzer senden oft die Passwort-E-Mail unmittelbar danach an denselben falschen Empfänger.
  • Beide E-Mails teilen das gleiche Irrtum-Risiko — der Fehler ist korreliert.

Einfach ausgedrückt: "Sie senden Schloss und Schlüssel an die falsche Person, ohne es zu bemerken."

④ Regierungen, Finanzregulierer und Großunternehmen haben PPAP explizit verboten

Japans Kabinett, Digitalagentur und Finanzdienstleistungsbehörde haben alle erklärt, dass das Versenden von passwortgeschützten ZIPs per E-Mail nicht empfohlen wird und haben Ministerien und Unternehmen aufgefordert, diese Praxis einzustellen.

"Der Versand von passwortgeschützten Dateianhängen sollte grundsätzlich nicht erfolgen."
— Japanische Finanzdienstleistungsbehörde (2025)

Zusammenfassung: Warum es gefährlich ist (einfache Sprache)

  1. Die Datei und ihr Passwort reisen denselben Weg — beide können abgefangen werden.
  2. Antivirus-Tools können nicht in verschlüsselte ZIPs scannen — Angreifer nutzen dies, um Malware zu verbreiten.
  3. Falsch gesendete Dateien werden mit ihren Schlüsseln gesendet, daher verhindert "Passwort später senden" die Offenlegung nicht.

Empfohlene Alternativen

  • Cloud-Speicher (Box / OneDrive / Google Drive) mit Freigabelinks
  • Verschlüsselte E-Mail (S/MIME / PGP)
  • Dedizierte sichere Dateiübertragungsdienste
  • Public-Key-Dateiverschlüsselung (dieses Tool)

Wie Public-Key-Kryptographie funktioniert

Das grundlegende Problem von PPAP war, dass Sie den Schlüssel (Passwort) an die andere Partei senden mussten. Public-Key-Kryptographie löst dies an der Wurzel.

1. Das Vorhängeschloss Analogie

Public-Key-Kryptographie verwendet ein Schlüsselpaar: einen öffentlichen Schlüssel und einen privaten Schlüssel.

  1. Schlüssel erstellen (Empfänger): Der Empfänger generiert ein "offenes Vorhängeschloss" (öffentlicher Schlüssel) und einen eindeutigen physischen Schlüssel (privater Schlüssel).
  2. Vorhängeschloss teilen: Der Empfänger gibt das offene Vorhängeschloss an jeden, der eine Datei senden möchte. Ein gestohlenes Vorhängeschloss ist ohne den Schlüssel nutzlos.
  3. Sperren und senden (Absender): Der Absender klickt das Vorhängeschloss auf der Datei zu. Nach dem Sperren kann nicht einmal der Absender es öffnen.
  4. Mit privatem Schlüssel entsperren (Empfänger): Der Empfänger verwendet seinen geheim gehaltenen privaten Schlüssel, um die Datei freizuschalten.

2. Warum es viel sicherer ist als PPAP

Der entscheidende Unterschied: Der Entschlüsselungsschlüssel (privater Schlüssel) reist nie über ein Netzwerk.

Kein Risiko, dass der Schlüssel gestohlen wird

Bei PPAP wird das Passwort (gemeinsamer Schlüssel) per E-Mail gesendet — dieser eine Abfangpunkt ist fatal. Bei der Public-Key-Kryptographie reist nur das "offene Vorhängeschloss" (öffentlicher Schlüssel) über das Netzwerk. Ein gestohlenes Vorhängeschloss ist völlig nutzlos zum Öffnen des gesperrten Inhalts.

Mathematische Einwegfunktion

Public-Key-Kryptographie basiert auf einer mathematischen Falltürfunktion — einfach in eine Richtung, rechnerisch unmöglich in die andere. Zum Beispiel ist das Multiplizieren zweier sehr großer Primzahlen augenblicklich, aber das Faktorisieren des Produkts (nur das Ergebnis gegeben) würde Supercomputer Jahre dauern.

3. Reale Anwendungen

  • HTTPS (Webbrowsing): Das Schlosssymbol in Ihrem Browser verwendet diesen Mechanismus zum Schutz von Daten.
  • S/MIME (E-Mail-Verschlüsselung): Von Universitäten und Regierungsbehörden verwendet, um Identitätswechsel zu verhindern und E-Mail-Inhalte zu schützen.
  • Digitale Signaturen: Der private Schlüssel signiert Daten; der öffentliche Schlüssel verifiziert sie — Authentizität ohne Offenlegung des Geheimnisses der Unterzeichner wird nachgewiesen.

Über dieses Tool (WebCrypto Exchange KH)

Dieses Tool implementiert die Public-Key-Dateiverschlüsselung vollständig im Browser.

Technische Hinweise

Web Crypto API: Das Tool verwendet window.crypto.subtle — eine Standard-Browser-API für Schlüsselgenerierung, Verschlüsselung und Entschlüsselung.

Lokale Dateiverarbeitung: Dateien werden mit der Blob / File API des Browsers gelesen und geschrieben. Dateiinhalte werden niemals an einen externen Server gesendet (es sei denn, Sie tun es selbst).

Statisches Hosting (GitHub Pages): Da GitHub Pages ein statischer Host ohne serverseitige Verarbeitung ist, kann die App technisch unmöglich Dateien oder Schlüssel auf einen Server hochladen.

Hybrid-Verschlüsselung

Das Tool verwendet Hybrid-Verschlüsselung:

  1. AES-256-GCM-Verschlüsselung: Ein zufällig generierter AES-Schlüssel verschlüsselt die Datei mit hoher Geschwindigkeit und unterstützt Dateien beliebiger Größe.
  2. RSA-OAEP-Schlüsselverschlüsselung: Der AES-Schlüssel wird dann mit dem öffentlichen Schlüssel des Empfängers verschlüsselt, sodass nur der Inhaber des privaten Schlüssels ihn wiederherstellen kann.

Dies macht es möglich, Dateien beliebiger Größe sicher zu verschlüsseln — Bilder, Videos, Archive und mehr.

Diese App ist ein leichtgewichtiges Tool und entspricht nicht der Robustheit von kommerziell oder organisatorisch verwalteten kryptographischen Lösungen.

Das heißt, es bietet eine messbare Sicherheitsverbesserung gegenüber PPAP (passwortgeschützte ZIP + Plaintext-Passwort-E-Mail). Wenn PPAP die Alternative ist, ist dieses Tool die bessere Wahl.
